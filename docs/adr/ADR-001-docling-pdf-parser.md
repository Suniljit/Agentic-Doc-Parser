# ADR-001: Docling for PDF Parsing

**Date:** 2026-05-18  
**Status:** Accepted  
**Deciders:** Sunil

---

## Context

The source document is the FY2024 Singapore Analysis of Revenue and Expenditure — a government financial PDF with three distinct content types:

- **Narrative text** — budget analysis, policy commentary
- **Tables** — revenue and expenditure breakdowns, year-on-year comparisons
- **Charts and figures** — fiscal position charts, revenue trend graphs (e.g. the page 8 fiscal balance chart from which a numeric value must be extracted)

All three parts of the pipeline depend on the parser's output:

| Part | Parser output used |
|------|-------------------|
| Part 1 — Structured Extraction | Pages 5, 6, 8, 20 (text + table + chart content) |
| Part 2 — Date Reasoning | Pages 1, 36 (text) |
| Part 3 — RAG Store | Full document (all content types chunked for embedding) |

A parser that misses any content type creates a blind spot. Specifically, if chart content is not captured, the fiscal position field (page 8) cannot be extracted and Part 1 fails an acceptance criterion.

---

## Alternatives Considered

### PyMuPDF (fitz)

Fast, mature library for text and basic layout extraction.

**Strengths:**
- Very fast (~1–2s for this PDF)
- Reliable text extraction and font metadata
- Basic table detection via block analysis

**Weaknesses:**
- No understanding of charts or figures — image regions are skipped or returned as empty blocks
- Table extraction is heuristic and fragile on complex layouts (merged cells, nested headers)

**Verdict:** Rejected. Chart blindness means page 8 fiscal position would be unextractable. Part 1 fails.

---

### pdfplumber

Python library built for structured table and text extraction.

**Strengths:**
- Excellent table extraction via character-level line analysis
- Handles complex column layouts better than PyMuPDF
- Widely used in financial document workflows

**Weaknesses:**
- Text and tables only — figures and charts produce no output
- Slower than PyMuPDF on large PDFs
- No markdown output natively (requires custom serialisation)

**Verdict:** Rejected. Same chart blindness as PyMuPDF. Also adds serialisation work with no benefit for this pipeline's LLM-ready markdown requirement.

---

### MarkItDown (Microsoft)

A format-conversion tool that outputs markdown from PDFs, Office documents, and other formats.

**Strengths:**
- Unified markdown output from many file types
- Simple one-call API

**Weaknesses:**
- Uses `pdfminer` under the hood for PDF extraction — text only
- No table or figure intelligence; inherits all limitations of text-only parsers
- No per-page access; the output is a flat markdown string

**Verdict:** Rejected. Combines the weaknesses of text-only parsers without any of Docling's multimodal capabilities.

---

## Decision

Use **Docling** (`docling` package, IBM Research) via `DocumentConverter`.

Docling performs a full layout analysis pass on the PDF using:
- A **table structure recognition model** (TableTransformer) to extract table content as structured data
- A **figure detection model** to identify chart regions and retain the image bytes
- Standard text extraction for narrative content

Chart descriptions are generated by **GPT-4o** via Docling's `PictureDescriptionApiOptions`, which posts each detected figure image to the OpenAI chat completions API. The description is stored in the `DoclingDocument`'s picture annotations and serialised into the JSON cache — GPT-4o is only called on the first parse.

All three content types are unified into a single `DoclingDocument` object and exported as markdown. This is the only evaluated option that captures chart content without requiring a separate post-processing step.

**Additional factors:**
- Native `export_to_markdown()` produces LLM-consumable output directly — no custom serialisation
- `DoclingDocument` exposes per-page content, enabling `parse_pages()` to slice specific pages without re-parsing
- JSON serialisation (`model_dump_json`) caches the entire document including chart descriptions; subsequent runs pay no parse or API cost

---

## Consequences

**Positive:**
- All three content types (text, tables, charts) captured in one pass
- Single parse, consistent markdown — Parts 1, 2, and 3 see identical content
- Chart descriptions from GPT-4o are accurate and quantitative (verified against the FY2024 PDF)
- Per-page access enables targeted extraction without re-parsing

**Negative:**
- First parse takes ~60–100s plus one GPT-4o vision call per chart (4 charts in this PDF)
- Heavier dependency footprint than text-only alternatives
- Chart descriptions are interpretive — GPT-4o may misread small axis labels; critical values should be verified against source

**Mitigations:**
- Full two-level cache (JSON disk + markdown disk) means the parse and API cost is paid exactly once
- `enable_remote_services=True` is an explicit opt-in, making the outbound call intentional and auditable
- Log parse start time and duration at INFO level so latency is visible and expected, not silent
